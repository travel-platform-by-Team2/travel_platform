# 03-API-명세-작성-가이드

## 이 스킬로 하는 일
기능 요구사항을 API 스펙 문서로 변환한다.

## 적용 대상
- 신규 API 추가
- 기존 API 리팩터링
- 프론트/백엔드 협업 인터페이스 고정

## 입력 정보
1. 엔드포인트 후보
2. Request DTO 구조
3. Response DTO 구조
4. 인증 필요 여부
5. 실패 시나리오

## 출력 결과
1. API 표(Method, URL, 설명, 상태)
2. 요청 예시/응답 예시
3. 검증 실패/권한 실패/미존재 자원 시나리오

## 작성 규칙
1. DTO 기준으로 필드를 설명한다.
2. 구현/예정 상태를 분리한다.
3. URL 규칙을 일관되게 유지한다.
4. 성공/실패 상태 코드를 명시한다.

## 권장 상태 코드
- 200: 조회 성공
- 201: 생성 성공
- 400: 입력 검증 실패
- 401: 인증 실패
- 403: 권한 없음
- 404: 리소스 없음
- 500: 서버 오류

## 주의사항
1. Controller 시그니처와 문서가 다르면 문서를 수정한다.
2. “추후 결정” 항목은 TODO로 남기고 위치를 명시한다.
3. 공개 API 변경은 승인 대상이다.

## 완료 기준
1. 프론트/백엔드가 동일 스펙으로 해석 가능
2. URL/메서드/필드 누락 없음
3. 테스트 케이스 작성에 바로 활용 가능
