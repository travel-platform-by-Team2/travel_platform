## 2. 스킬 트리거 규칙

요청에 아래 키워드/의도가 포함되면 해당 스킬을 우선 적용한다.

1. 도메인 뼈대 생성/신규 엔티티 시작: `01`
2. 단계적 개발 계획, V1/V2/V3 분할: `02`
3. API 문서/명세/응답 예시 작성: `03`
4. 샘플 데이터/`data.sql`/시딩: `04`
5. 로그인/로그아웃/세션 헤더: `05`
6. 코드 변경 후 문서 갱신/작업로그 반영: `06`
7. 테스트 시나리오/회귀 테스트 추가: `07`
8. 예외 처리 통일/상태코드 매핑: `08`
9. 화면 라우트/뷰 경로/404 점검: `09`
10. 느린 조회/N+1/쿼리 로그 분석: `10`
11. 본인 데이터만 접근/401/403 정책: `11`
12. 테스트 작업 실행 표준/결과 보고: `12`
13. 코드 스타일 통일/코드 컨벤션 수립/일괄 리팩토링: `13`
14. Java 코드 생성/수정 시 컨벤션 강제 적용: `14`

복수 스킬 적용 규칙:

1. 먼저 계획 성격 스킬(`02`, `03`)을 적용한다.
2. 구현 성격 스킬(`01`, `04`, `05`, `08`, `11`, `13`, `14`)을 적용한다.
3. 검증/정리 스킬(`07`, `09`, `10`, `12`, `06`)로 마무리한다.

## 3. 공통 입력/출력 계약

### 3.1 입력 계약

모든 스킬은 아래 입력을 공통으로 확인한다.

1. 작업 목표(한 문장)
2. 작업 범위(포함/제외)
3. 위험도(LOW/MEDIUM/HIGH)
4. 승인 필요 여부(RULES 기준)
5. 관련 파일/경로

### 3.2 출력 계약

모든 스킬 결과는 아래 형식을 따른다.

1. 작업 요약
2. 변경 파일 목록(추가/수정/삭제)
3. 검증 결과(명령/결과)
4. TODO
5. 리스크/주의사항

### 3.3 중단 조건

아래 조건이면 즉시 중단 후 사용자 재확인을 요청한다.

1. Entity 구조 변경 필요
2. DB schema 변경 필요
3. 인증/권한 로직 변경 필요
4. 공개 API 스펙 변경 필요
5. `application.properties` 변경 필요
6. 대규모 변경(파일 5개 이상 또는 diff 200줄 이상)

### 3.4 컨트롤러 타입 결정 규칙

1. JSON 데이터 송수신 목적의 컨트롤러는 클래스 레벨 `@RequestMapping` 경로에 `/api`를 포함한다.
2. 1번 조건의 도메인은 `@RestController`를 사용한다.
3. 1번 조건이 아닌 화면 렌더링 목적(`return "pages/..."`) 도메인은 `@Controller`를 사용한다.
4. `@RestController`는 데이터 응답 전용이며 뷰 이름(템플릿 경로 문자열)을 반환하지 않는다.
5. 동일 도메인에서 View와 API가 공존하면 컨트롤러 클래스를 반드시 분리한다.
6. 클래스 명명은 `<Domain>Controller`(View), `<Domain>ApiController`(JSON)를 사용한다.
7. `@RequestBody`는 JSON 요청 본문 입력에만 사용한다.
8. JSON 응답은 `@RestController` 또는 `@ResponseBody`/`ResponseEntity`로 처리한다.
9. `GET` 메서드에서는 `@RequestBody` 대신 `@RequestParam`/`@PathVariable`을 사용한다.
10. `01-도메인-스캐폴딩`, `09-라우팅-뷰매핑-점검`, `13-코드컨벤션-리팩토링`, `14-자바-코드컨벤션-적용` 스킬 적용 시 위 규칙을 필수 점검한다.

## 4. 스킬 실행 플로우

1. 요청 분류

- 작업 유형 선언: 기능 추가/버그 수정/리팩토링/테스트 보강/문서 수정/설정 변경
- 적용 스킬 후보 1~2개 선정

2. 범위 확정

- 포함/제외 범위 명시
- 승인 필요 작업 여부 판정

3. 스킬 실행

- 선택한 스킬 문서의 입력 항목을 채우고 결과물을 작성/구현
- RULES의 계층/금지 규칙 준수

4. 검증

- 최소 검증 수행(컴파일/테스트/문서 정합성)
- 실패 시 원인과 재현 조건 기록

5. 보고 및 동기화

- 결과 보고 형식으로 산출물 정리
- 필요 시 `06-문서동기화-업데이트` 스킬로 문서 반영

## 5. 테스트 작업 운영 기준

테스트 작업은 `12-테스트-작업-운영` 스킬을 기준으로 수행한다.

참조 스킬:

- `.docs/skills/12-테스트-작업-운영/12-테스트-작업-운영-가이드.md`

적용 원칙:

1. 테스트 작업 요청 시 `12` 스킬을 우선 적용한다.
2. 구조/권한/API 변경이 포함되면 `.docs/RULES.md` 승인 규칙을 먼저 확인한다.
3. 테스트 결과는 명령어와 성공/실패를 분리해 기록한다.

실행 기준:

1. 테스트 범위를 정의한다(단위/통합/회귀).
2. 시나리오를 구성한다(정상/검증실패/권한실패/예외처리).
3. 테스트를 실행하고 실패 시 재현 조건을 기록한다.
4. 결과를 보고 형식(변경 파일/검증 결과/TODO/리스크)으로 정리한다.

문서 유지 규칙:

1. 테스트 작업 운영 기준 변경 시 본 문서를 갱신한다.
2. 실제 테스트 절차 변경 시 `12-테스트-작업-운영-가이드.md`를 먼저 갱신한다.
