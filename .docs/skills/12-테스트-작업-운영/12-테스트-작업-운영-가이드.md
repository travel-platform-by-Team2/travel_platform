# 12-테스트-작업-운영-가이드

## 이 스킬로 하는 일
테스트 작업 요청을 받아 시나리오 설계, 테스트 작성/실행, 결과 보고까지 일관된 절차로 수행한다.

## 적용 대상
- 신규 기능 개발 직후 테스트 보강
- 버그 수정 후 회귀 테스트
- 릴리즈 전 핵심 시나리오 검증
- 기존 테스트 실패 원인 분석

## 입력 정보
1. 작업 목표(테스트하려는 기능/버그)
2. 대상 범위(도메인, API, 서비스, 화면)
3. 관련 정책(인증/권한/예외 응답)
4. 완료 기준(통과 조건)
5. 실행 가능한 테스트 명령

## 출력 결과
1. 테스트 시나리오 목록(정상/실패/권한/예외)
2. 테스트 코드 변경 내역(추가/수정/삭제)
3. 실행 결과(명령/성공 여부/실패 원인)
4. TODO 및 리스크

## 작업 절차
1. 테스트 범위를 확정한다.
2. 최소 시나리오 세트를 작성한다.
3. `src/main`과 동일한 패키지 구조로 `src/test`에 테스트 코드를 추가/수정한다.
4. 테스트를 실행하고 실패 원인을 기록한다.
5. 결과를 표준 보고 형식으로 정리한다.

## 테스트 폴더 구조 규칙
1. 테스트 코드는 `src/test/java`에서 작성한다.
2. 패키지 구조는 `src/main/java`와 동일하게 유지한다.
3. 테스트 클래스명은 `대상 + Test`, 통합 테스트는 `*IntegrationTest`를 사용한다.

## 최소 시나리오 세트
1. 정상 케이스 1개 이상
2. 검증 실패 케이스 1개 이상
3. 권한 실패 케이스 1개 이상(해당 시)
4. 예외 처리 케이스 1개 이상

## 검증 명령 예시
- `./gradlew test`
- `./gradlew test --tests *도메인명*`
- `./gradlew test --tests *IntegrationTest`

## 보고 형식
1. 작업 요약
2. 변경 파일 목록(추가/수정/삭제)
3. 검증 결과(명령/결과)
4. 미완료 항목(TODO)
5. 리스크/주의사항

## 주의사항
1. 테스트 없는 핵심 로직 변경을 허용하지 않는다.
2. 스펙과 불일치하는 테스트 통과는 실패로 본다.
3. flaky 테스트를 추가하지 않는다.
4. 승인 대상 변경(Entity/권한/API/설정) 포함 시 사용자 승인 후 진행한다.

## 완료 기준
1. 최소 시나리오 세트를 충족한다.
2. `./gradlew test` 또는 합의된 대상 테스트가 통과한다.
3. 실패 케이스의 원인과 재현 조건이 기록된다.
4. 결과 보고 형식이 누락 없이 작성된다.
