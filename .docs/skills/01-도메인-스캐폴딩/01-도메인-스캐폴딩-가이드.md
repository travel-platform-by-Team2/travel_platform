# 01-도메인-스캐폴딩-가이드

## 이 스킬로 하는 일
새 기능을 시작할 때, 도메인 관련 기본 파일들을 한 번에 일관된 형태로 만든다.

## 참조 문서
- `.docs/specs/java-code-convention.md`
- `.docs/RULES.md`

## 적용 대상
- 신규 도메인 추가
- 기존 도메인 확장(필드/관계 추가)
- API 구현 전 뼈대 작업

## 입력 정보
1. 도메인 이름(예: Booking, TripPlan)
2. 테이블 이름
3. 주요 필드(타입, nullable 여부)
4. 관계(FK, 다대일/일대다)
5. 기본 URL prefix(예: `/bookings`)
6. 라우트 성격(View 렌더링 / JSON API)

## 생성 결과
1. Entity
2. Repository
3. Service
4. Page Controller(필요 시)
5. API Controller(필요 시)
6. Request DTO
7. Response DTO

## 생성 규칙
1. 패키지는 기능 단위로 분리한다.
2. 의존 방향은 Controller -> Service -> Repository 고정.
3. 내부 로직은 TODO 중심으로 두고, 시그니처만 확정한다.
4. DTO와 Entity를 섞지 않는다.
5. `@Data`, `@NoArgsConstructor` 사용 기준을 통일한다.
6. JSON 데이터 송수신 목적 도메인은 클래스 레벨 `@RequestMapping("/api/...")` + `@RestController`를 사용한다.
7. JSON 데이터 송수신 목적이 아닌 화면(View) 렌더링 도메인은 `@Controller`를 사용한다.
8. View 라우트와 API 라우트는 컨트롤러 클래스를 분리한다.
9. 클래스 명명은 `<Domain>Controller`, `<Domain>ApiController` 패턴을 사용한다.
10. JSON 요청 본문 입력이 있는 API 메서드는 `@RequestBody`(필요 시 `@Valid`)를 사용한다.
11. `GET` API는 `@RequestBody`를 사용하지 않고 `@RequestParam`/`@PathVariable`을 사용한다.
12. `@RestController` 메서드는 뷰 경로 문자열을 반환하지 않고 DTO/`ResponseEntity` 등 데이터만 반환한다.

## 작업 절차
1. 유즈케이스를 `View 라우트`와 `API 라우트`로 먼저 분류한다.
2. Entity 먼저 작성하고 컬럼/관계를 고정한다.
3. Repository에 최소 메서드 시그니처를 만든다.
4. Service에 유즈케이스 단위 메서드를 만든다.
5. 라우트 성격에 맞게 컨트롤러를 연결한다(`JSON=@RequestMapping("/api/...")+@RestController`, `그 외=@Controller`).
6. JSON 입력이 필요한 API 메서드는 `@RequestBody` + Request DTO를 연결한다.
7. Request/Response DTO를 유즈케이스 기준으로 분리한다.

## 주의사항
1. 인증/권한 로직은 임시값으로 고정하지 말고 TODO로 명시한다.
2. Entity 구조 변경은 고위험 작업으로 보고 승인 후 진행한다.
3. Controller에서 Repository 직접 호출 금지.
4. 컨트롤러 타입과 라우트 목적이 불일치하면 구현을 중단하고 규칙부터 재확인한다.

## 완료 기준
1. 컴파일 성공
2. 테스트 실행 성공(`./gradlew test`)
3. 라우트/DTO/서비스 메서드 이름이 기능 의미와 일치
4. 컨트롤러 타입 규칙(`JSON=@RequestMapping("/api/...")+@RestController`, `그 외=@Controller`)을 만족
5. `@RequestBody` 사용 규칙(`요청 본문 전용`, `GET 미사용`)을 만족
6. `@RestController`에서 뷰 경로 반환이 없다.
