# 13-코드컨벤션-리팩토링-가이드

## 이 스킬로 하는 일
팀원 코드 병합 이후 불규칙한 코드 스타일/구조를 공통 규칙으로 정리하고, 코드 컨벤션 문서를 기준으로 리팩토링을 수행한다.

## 참조 문서
- `.docs/specs/java-code-convention.md`
- `.docs/RULES.md`

## 적용 대상
- 팀원 코드 병합 후 네이밍/포맷/구조가 불일치한 경우
- 도메인별 코딩 스타일이 달라 유지보수 비용이 커진 경우
- 리팩토링 전에 공통 코드 컨벤션 문서가 없는 경우
- 규칙 기반 일괄 정리가 필요한 경우

## 입력 정보
1. 리팩토링 대상 범위(패키지/도메인/파일 목록)
2. 기준 브랜치와 비교 범위(예: `main..feature/*`)
3. 언어/프레임워크 기준(Java/Spring, Mustache, JS 등)
4. 필수 컨벤션 항목(네이밍, 들여쓰기, 계층 규칙, DTO/Entity 분리, 컨트롤러 타입 규칙)
5. 자동 정리 도구 사용 범위(포맷터/정적분석/IDE 정리)
6. 승인 필요 작업 여부(RULES 기준)

## 출력 결과
1. 코드 컨벤션 문서 정합성 확인/갱신 (`.docs/specs/java-code-convention.md`)
2. 불일치 항목 분류표(치명/일반/권장)
3. 리팩토링 적용 결과(변경 파일 목록, 적용 규칙)
4. 검증 결과(빌드/테스트/정적 점검)
5. 후속 작업 TODO(이번 범위 제외 항목)

## 컨벤션 고정 규칙
1. JSON 데이터 송수신 목적의 컨트롤러는 클래스 레벨 `@RequestMapping` 경로에 `/api`를 포함한다.
2. 1번 조건의 도메인은 `@RestController`를 사용한다.
3. 1번 조건이 아닌 도메인은 `@Controller`를 사용한다.
4. View 반환과 JSON 응답은 같은 컨트롤러에서 혼용하지 않는다.
5. 동일 도메인에서 View/API가 공존하면 `{Domain}Controller`, `{Domain}ApiController`로 클래스를 분리한다.
6. `@RequestBody`는 JSON 요청 본문 입력 메서드에만 사용한다.
7. JSON 응답은 `@RestController` 또는 `@ResponseBody`/`ResponseEntity`로 처리한다.
8. `GET` 메서드에서는 `@RequestBody`를 사용하지 않는다.
9. `@RestController` 메서드는 뷰 경로 문자열을 반환하지 않는다.

## 작업 절차
1. 대상 코드에서 반복 불일치 패턴을 수집한다(네이밍, 포맷, 계층, 예외 처리, 주석 규칙).
2. 불일치 항목을 치명/일반/권장으로 분류하고 이번 작업 범위를 확정한다.
3. `.docs/specs/java-code-convention.md` 기준으로 규칙 충돌/누락을 먼저 정리한다.
4. 컨트롤러를 View/API 기준으로 재분류하고 `{Domain}Controller`/`{Domain}ApiController` 분리 리팩토링을 우선 수행한다.
5. API 메서드에서 `@RequestBody`/`@RequestParam`/`@PathVariable` 사용이 규칙에 맞는지 정리한다.
6. 자동화 가능한 항목(포맷, import 정리, 단순 네이밍)을 일괄 정리한다.
7. 수동 판단이 필요한 항목(메서드 분리, 책임 이동, 네이밍 재설계)을 순차 리팩토링한다.
8. 동작 회귀 검증(컴파일/테스트/핵심 시나리오)을 수행한다.
9. 결과 보고 형식에 맞춰 변경 내역, 검증 결과, TODO를 기록한다.

## 주의사항
1. 리팩토링은 동작 변경이 아닌 구조/가독성 개선에 한정한다.
2. 공개 API/DB schema/인증·권한/설정 파일 변경은 RULES 승인 없이 진행하지 않는다.
3. 대규모 변경(파일 5개 이상 또는 diff 200줄 이상)은 작업 전 승인 기준을 다시 확인한다.
4. 팀 합의 없는 신규 규칙을 임의로 강제하지 않고, 문서에 근거를 남긴다.
5. 자동 정리 도구 적용 시 범위를 명확히 제한하고 결과 diff를 검토한다.

## 완료 기준
1. `.docs/specs/java-code-convention.md`와 실제 코드가 모순 없이 일치한다.
2. 대상 범위의 치명 등급 불일치 항목이 0건이다.
3. 리팩토링 변경 파일과 적용 규칙이 추적 가능하게 기록되어 있다.
4. `./gradlew test` 또는 동등한 검증 절차가 성공했다.
5. 후속 TODO와 리스크가 문서에 명시되어 있다.
