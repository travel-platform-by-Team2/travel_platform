## 2. 우선 참조 문서

1. `.docs/RULES.md`
2. `.docs/.agent/TASK_TEMPLATE.md`
3. 프로젝트 운영 문서 (`.docs/.agent/*.md`)
4. 기능별 스킬 문서 (`.docs/skills/*`)

## 3. 실행 절차

1. 요청 분석
2. 작업자 폴더명 확인 (미지정 시 사용자에게 먼저 확인)
3. TASK 문서 우선 작성 (`TASK_TEMPLATE.md` 기준, 경로: `.docs/.task/.{작업자폴더}/flow`)
4. TASK 문서에 작업 Workflow 문서화 (단계, 입출력, 검증, 중단 조건)
5. 범위 확정(포함/제외) 및 위험도 판정(LOW/MEDIUM/HIGH)
6. 승인 필요 여부 확정
7. 생성된 TASK 문서를 기준으로 구현/문서 작업 수행
8. 검증 수행
9. 결과 보고 및 TASK 결과 기록 업데이트

## 4. 의사결정 규칙

- 우선순위: 사용자 지시 > `.docs/RULES.md` > 프로젝트 정책 > 개별 에이전트 규칙
- 모호한 요구는 추정하지 않고 질문으로 확정
- 범위 외 변경 필요 시 작업 중단 후 재합의
- 작업 지시를 받으면 코드/문서 수정 전에 TASK 문서를 먼저 생성한다
- TASK 문서에 정의되지 않은 작업은 임의로 수행하지 않는다
- 작업자 폴더명이 없으면 TASK 생성/작업 수행 전에 사용자에게 폴더명을 질문한다

## 5. 승인 필요 작업

- Entity 구조 변경
- DB schema 변경
- 인증/권한 로직 변경
- 공개 API 스펙 변경
- 설정 파일(`application.properties`) 변경
- 대규모 변경(파일 5개 이상 또는 diff 200줄 이상)

## 6. 금지 규칙

- Controller -> Repository 직접 호출
- DTO/Entity 역할 혼합
- 테스트 없는 핵심 로직 변경
- 사용자 확인 없는 파괴적 명령 실행
