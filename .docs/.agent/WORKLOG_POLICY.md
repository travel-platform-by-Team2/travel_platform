# WORKLOG_POLICY.md

이 문서는 작업일지 작성 기준과 실행 명령을 정의한다.
작업일지는 `.docs/.task` 하위의 개인 폴더에 누적 기록한다.

관련 문서:
- `.docs/skills/00-스킬인덱스/00-스킬인덱스.md`
- `.docs/.agent/TASK_TEMPLATE.md`
- `.docs/skills/12-테스트-작업-운영/12-테스트-작업-운영-가이드.md`

---

## 1. 목적

1. 사용자 요구사항과 실제 작업 내역을 같은 문서에서 추적한다.
2. 변경 파일과 테스트 결과를 일자 단위로 빠르게 확인한다.
3. 다음 작업자가 문서만 보고 현재 상태를 이해할 수 있게 한다.

---

## 2. 생성 위치 및 파일명

1. 작업일지 위치: `.docs/.task/.{개인폴더명}`
2. 파일명: `TASK_WORKLOG.md` (고정)
3. 새 작업은 파일 하단에 `명령` 섹션으로 추가한다.
4. 팀원 간 같은 파일을 공유하지 않는다.

예시:

```text
.docs/.task/.khj/TASK_WORKLOG.md
.docs/.task/.abc/TASK_WORKLOG.md
```

---

## 3. 작업일지 필수 항목

1. 사용자 요구사항 요약
2. 작업 진행 내용
3. 변경 파일 목록
4. 파일별 변경 유형
5. 테스트 결과
6. TODO / 리스크

---

## 4. 명령 섹션 규칙

1. 개인 폴더의 단일 파일에 명령 단위로 누적한다.
2. 명령 번호는 `001`, `002`, `003` 순서로 증가한다.
3. 섹션 제목 형식은 아래를 따른다.

```text
## 명령 001 - 2026-03-03
```

---

## 5. 변경 파일 기록 규칙

1. 변경 유형은 `추가`, `수정`, `삭제` 중 하나로 기록한다.
2. 파일 경로는 프로젝트 루트 기준 상대 경로로 기록한다.
3. 파일이 없으면 `변경 파일 없음`으로 명시한다.

기록 예시:

```text
- 수정: src/main/java/com/example/auth/AuthService.java
- 추가: src/test/java/com/example/auth/AuthServiceTest.java
- 삭제: src/main/resources/old-template.mustache
```

---

## 6. 테스트 결과 기록 규칙

1. 실행한 명령을 그대로 적는다.
2. 결과는 `성공`, `실패`, `미실행`으로 명확히 적는다.
3. 실패 시 원인 요약과 재현 조건을 함께 적는다.

---

## 7. 작업일지 작성 명령

아래 명령으로 작업일지 작성을 요청한다.

```text
[작업일지 작성 요청]
- 개인 폴더명: khj
- 명령 번호: 001
- 날짜: 2026-03-03
- 사용자 요구사항:
- 작업 진행 내용:
- 변경 파일:
- 테스트 결과:
- TODO/리스크:

위 내용을 기준으로 .docs/.task/.khj/TASK_WORKLOG.md에 새 명령 섹션을 추가해줘.
변경 파일은 추가/수정/삭제로 구분해줘.
테스트 결과는 명령어와 성공/실패 여부를 함께 적어줘.
```
